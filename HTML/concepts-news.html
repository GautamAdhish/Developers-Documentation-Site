What Can You Build? — ASOBI Game Portal

A real project built with only HTML, CSS, and JavaScript — and a breakdown of the parts that beginners usually find confusing.

ASOBI — Mini Game Portal

A fully functional browser-based game portal featuring Snake, Rock Paper Scissors, Tic Tac Toe and more. Built entirely with HTML, CSS, and vanilla JavaScript — no frameworks, no libraries. This is exactly the kind of project you can build once you know the fundamentals.

CSS Concept — Animations and Staggered Delays

The bouncing letters on the ASOBI welcome screen look like a complex effect but they are made from two simple CSS tools: @keyframes and animation-delay. Understanding these two things lets you build almost any animation you can imagine.

@keyframes is where you define the motion itself. You describe what the element looks like at the start, the middle, and the end of the animation. The browser fills in everything in between. Once defined, you attach the animation to any element using the animation property.

How @keyframes works

This is the exact animation used in ASOBI. The letter starts at its normal position, moves 20px upward at the halfway point, then returns back down.

@keyframes bounce { 0%, 100% { transform: translateY(0); } /* start and end in original position */ 50% { transform: translateY(-20px); } /* at halfway, jump 20px upward */ } .welcome span { animation: bounce 1s infinite; /* name | duration | repeat forever */ } /* stagger each letter with a slightly bigger delay */ .welcome span:nth-child(1) { animation-delay: 0s; } .welcome span:nth-child(2) { animation-delay: 0.1s; } .welcome span:nth-child(3) { animation-delay: 0.2s; } .welcome span:nth-child(4) { animation-delay: 0.3s; } .welcome span:nth-child(5) { animation-delay: 0.4s; }

Without the delays all five letters would bounce at exactly the same moment and look like one solid block. By giving each letter a slightly larger delay the animation starts at a different time for each one, creating the wave effect. :nth-child(n) is a CSS selector that targets an element by its position number inside its parent.

CSS Concept — Custom Fonts and Fluid Font Sizing

ASOBI uses a pixel font called "Minecraft" that is not available on Google Fonts. The font file lives inside the project folder and is loaded manually using @font-face. This is a standard CSS rule that tells the browser about a font file you are hosting yourself rather than pulling from an external source.

On top of that, the welcome text scales smoothly across screen sizes without using media queries. This is done using clamp() — a CSS function that lets you define a minimum size, a preferred size, and a maximum size all in one value.

@font-face and clamp()
/* load the font file from your own project folder */ @font-face { font-family: "Minecraft"; /* the name you'll use in CSS from now on */ src: url("./assets/fonts/Minecraft.ttf") format("truetype"); } .welcome { font-size: clamp(2.5rem, 8vw, 20rem); /* min preferred max */ }

The font-family value is a name you make up — you use it later anywhere you write font-family: "Minecraft". The src is the path to the font file stored in your project. For clamp, 2.5rem is the smallest the text will ever be, 8vw is the fluid preferred size (8% of the browser window width), and 20rem is the maximum it can grow to. Without clamp you would need several media query breakpoints just to handle font scaling.

CSS Concept — Glowing Background and Sticky Navbar

The dark background that glows purple in the centre is not an image — it is a single line of CSS using radial-gradient. The navbar that stays fixed at the top as you scroll uses position: sticky combined with z-index to make sure nothing overlaps it. Both of these are patterns you will reuse in almost every project.

radial-gradient and position: sticky
/* deep purple glow in the centre, fades to black at the edges */ body { background: radial-gradient(circle, #1a002b, #000); } .navbar { position: sticky; /* sticks when you scroll past it */ top: 0; /* sticks to the very top */ z-index: 9; /* sits above all other content */ background: rgba(10, 0, 20, 0.8); /* semi-transparent */ }

A radial gradient radiates outward from a centre point. circle sets its shape, #1a002b is the dark purple inner colour, and #000 is the pure black outer edge. For the navbar, z-index controls stacking order — a higher number means closer to the front, so page content scrolls underneath it rather than on top. rgba(10, 0, 20, 0.8) is a colour with transparency — the 0.8 makes it 80% opaque so content behind it slightly shows through.

CSS Concept — Responsive Menu with @media and transform

On desktop the nav links sit in a row. On mobile they collapse behind a hamburger button. The switch is entirely CSS — a @media query that only applies below 768px wide. Inside it, the nav is hidden using three properties together, and the hamburger icon transforms into an X using transform: rotate().

@media query and hamburger → X transform
@media (max-width: 768px) { .nav-links { opacity: 0; /* invisible */ visibility: hidden; /* removed from interaction */ pointer-events: none; /* unclickable */ transform: translateY(-20px); /* shifted up */ transition: all 0.3s ease; } .nav-links.active { opacity: 1; visibility: visible; pointer-events: all; transform: translateY(0); /* slide back into place */ } /* top line rotates to form one arm of the X */ .hamburger.active span:nth-child(1) { transform: rotate(45deg) translate(9px, 9px); } .hamburger.active span:nth-child(2) { opacity: 0; } .hamburger.active span:nth-child(3) { transform: rotate(-45deg) translate(8px, -8px); } }

Three properties are needed to fully hide the nav — opacity: 0 alone makes it invisible but still clickable, causing accidental taps. visibility: hidden removes it from interaction. pointer-events: none is a safety net. All three together allow the transition to animate the reveal smoothly when .active is added. The translate inside the rotate moves each line so the two arms cross exactly at the centre.

JavaScript Concept — Event Listeners and classList

The hamburger menu is controlled entirely by JavaScript adding and removing a CSS class called active. The CSS already knows what to do when that class is present — JavaScript just decides when to add or remove it. This separation of concerns is a core pattern in front-end development: CSS handles appearance, JavaScript handles behaviour.

Three separate event listeners work together: one on the hamburger button to toggle the menu open and closed, one on each nav link to close the menu when a link is clicked, and one on the entire document to close the menu when the user clicks anywhere outside it.

classList.toggle and stopPropagation
const hamburger = document.getElementById("hamburger"); const navLinks = document.getElementById("navLinks"); hamburger.addEventListener("click", function(e) { e.stopPropagation(); /* stops the click from also firing the document listener below */ hamburger.classList.toggle("active"); navLinks.classList.toggle("active"); /* toggle adds the class if absent, removes it if present */ }); /* close when clicking outside */ document.addEventListener("click", function(e) { if (!hamburger.contains(e.target) && !navLinks.contains(e.target)) { hamburger.classList.remove("active"); navLinks.classList.remove("active"); } });

e.stopPropagation() is needed because clicks bubble upward — a click on the hamburger also counts as a click on the document. Without this line, clicking the hamburger would open the menu and the document listener would immediately close it again. .contains(e.target) checks whether the element that was clicked exists inside a given parent — it returns true or false. The ! flips the result, so the condition reads "if the click was NOT inside the hamburger AND NOT inside the nav, close the menu."

querySelectorAll and closing on link click
const links = navLinks.querySelectorAll("a"); /* selects every <a> tag inside the nav — returns a list */ for (let i = 0; i < links.length; i++) { links[i].addEventListener("click", function() { hamburger.classList.remove("active"); navLinks.classList.remove("active"); }); }

querySelectorAll returns a NodeList — similar to an array — of every element matching the CSS selector. The for loop walks through each one and attaches the same click listener to all of them. links[i] accesses the item at position i — indexes start at 0, so links[0] is the first link, links[1] is the second, and so on. This is far cleaner than writing a separate addEventListener call for every single link.
